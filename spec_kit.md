▸ Какие связи между сущностями?
1.	Company -> Certificate
•	Тип: один-ко-многим (1:N)
•	Обоснование: одна компания может выпустить много сертификатов. Каждый сертификат принадлежит ровно одной компании.
•	FK: Certificate.company_id -> Company.id
2.	Company -> Alert
•	Тип: один-ко-многим (1:N)
•	Обоснование: каждая компания получает множество уведомлений о подделках своих сертификатов.
•	FK: Alert.company_id -> Company.id
3.	Certificate -> VerificationRequest
•	Тип: один-ко-многим (1:N)
•	Обоснование: один и тот же оригинальный сертификат может быть «похож» на множество подозрительных PDF (разные подделки). Каждый запрос проверки ссылается на один найденный сертификат (или ни на один, если совпадений нет).
•	FK: VerificationRequest.matched_certificate_id -> Certificate.id
4.	Certificate -> Alert
•	Тип: один-ко-многим (1:N)
•	Обоснование: один сертификат может породить несколько уведомлений – 
	если разные проверки обнаруживают подделки, похожие на него.
•	FK: Alert.certificate_id -> Certificate.id
5.	VerificationRequest -> Alert
•	Тип: один-к-одному (1 : 1) – или один-ко-многим, но логичнее один-к-одному
•	Обоснование: каждый запрос проверки, который дал положительный результат (сходство выше порога), порождает одно уведомление. Повторные уведомления по тому же запросу не нужны.
•	FK: Alert.verification_request_id -> VerificationRequest.id (уникальное ограничение)

── [02] ГРАНИЦЫ ЗНАЧЕНИЙ
Что происходит на краях диапазона.
▸ Что при пустом состоянии? (0 элементов, пустая строка, пустой массив)
1.	Пустой список / 0 элементов
•	0 компаний в системе -> запрос GET возвращает []
•	0 сертификатов у компании -> GET возвращает []
•	0 запросов на проверку -> GET возвращает []
•	В базе нет ни одного сертификата (0 векторов) -> при попытке проверить подозрительный PDF:
o	Система должна вернуть [].
o	Уведомление не отправляется.
2.	Обучение модели TF-IDF
•	Пустой корпус текстов (0 сертификатов) -> при пустом корпусе не вызывать fit, GET возвращает [].
3.	Пустая строка ("" или null)
•	Company.name = "" -> Нет, обязательное поле (NOT NULL, длина ≥ 1). При Create возвращать 400.
•	Company.email = "" -> Нет, обязательное поле, валидация формата.
•	Certificate.recipient_name = "" -> Необязательным, запретить пустую строку, defulte = "Unknown".
•	Certificate.certificate_number = ""	 -> уникальный идентификатор, обязательное.
•	Certificate.pdf_path = "" -> Нет, обязательное поле, указывает на реальный файл.
•	Certificate.hash_sha256 = "" -> Нет, обязательное поле, всегда вычисляется.
•	Certificate.tfidf_vector = [] Нет, обязательное поле, при создании вычисляется автоматически, не может быть пустым.
•	VerificationRequest.suspicious_file_path = "" -> Нет, обязательное поле, всегда указывает на загруженный PDF.
•	VerificationRequest.matched_certificate_id = null -> Да, нормальное состояние для ненайденного оригинала.
•	Alert.message_text = "" -> Нет, уведомление должно иметь текст, генерируется автоматически.
4.	Извлечение текста из PDF
•	PDF пустой (0 страниц или страницы без текста) -> при получении пустого текста сразу возвращать, сообщить пользователю о некорректном вводе.
▸ Что при максимальных значениях? (длинная строка, большое число, много элементов)
1.	Длинные строки
•	Company.name -> VARCHAR(255)	Ошибка валидации (400 Bad Request)
•	Company.email -> VARCHAR(320) Ошибка валидации (400 Bad Request)
•	Certificate.recipient_name -> VARCHAR(255) Ошибка валидации (400 Bad Request)
•	Certificate.certificate_number -> VARCHAR(36) Контролируется системной
•	pdf_path -> VARCHAR(512) Контролируется системной
2.	Большие числа
•	similarity_score -> FLOAT (0..1) Контролируется системной
•	id -> (BIGINT) В учебном проекте не достижимо.
•	issue_date -> DATE/TIMESTAMP	При вводе будущих дат – валидация: дата выдачи не может быть в будущем.
3.	Много признаков в TF-IDF (max_features)
•	max_features=5000 и явно указать это в конфиге.
4.	Много запросов на проверку (VerificationRequest)
•	Ограничение 1 ГБ или удалять старые файлы еженедельно. Добавить механизм архивации (перемещать записи старше 5 дней в архивную таблицу).
5.	Много уведомлений (Alert)
•	Архивировать вместе с VerificationRequest
6.	Максимальный размер загружаемого PDF
•	Ограничение на уровне веб-сервера (nginx) и приложения: 10 МБ.
7.	Максимальное количество одновременно обрабатываемых запросов (concurrency)
•	Не более 10-20 параллельных проверок, иначе память и CPU будут перегружены.
── [03] ОБРАБОТКА ОШИБОК И СБОЕВ
Что происходит, когда что-то идёт не так.
▸ Что при отказе внешней зависимости? (сеть, БД, внешний 
1.	Основными внешними зависимостями являются:
•	База данных (PostgreSQL)
•	Файловая система (хранение загруженных PDF)
•	Сеть / внешние API (отправка email)
•	Очередь задач (RabbitMQ) — для фоновой обработки VerificationRequest
2.	Отказ базы данных
•	Использовать connection pooling с таймаутами.
•	Оборачивать все операции с БД в try/except
•	Возвращать HTTP 503 Service Unavailable с понятным сообщением: “База данных временно недоступна, попробуйте позже”.
•	Логировать ошибку с уровнем CRITICAL.
3.	Отказ файловой системы (недоступно место для загрузки PDF)
•	Перед записью проверять доступное место Если свободно < 15 МБ – отклонять загрузку.
•	При ошибке записи – возвращать 500 Internal Server Error или 507 Insufficient Storage.
•	Логировать с указанием пути и причины.
4.	Отказ внешнего API (отправка email)
•	Отправку уведомлений делать асинхронно (через фоновую задачу или очередь).
•	При первой ошибке логировать, но не падать.
•	Если доставка не удалась – перевести Alert в состояние delivery_status = 'failed'.
•	Предусмотреть retry (повторные попытки с экспоненциальной задержкой, максимум 3-5 раз).
•	Если после всех попыток не удалось – оставить failed и, возможно, записать в отдельную таблицу ошибок для ручного вмешательства администратора.
5.	Отказ модели TF-IDF / NearestNeighbors
•	При старте приложения всегда пробовать загрузить модель. 
▸ Что при невалидном вводе?
1. Company (создание / обновление)
name — пустая строка {"name": ""} -> 400 "Название компании обязательно"
name — превышает длину (255) 300 символов -> 400 "Название не должно превышать 255 символов"
email — неверный формат	"company@" или "mail.ru" -> 400 "Укажите корректный email адрес"
email — пустой "" -> 400 "Email обязателен для уведомлений"
email — дубликат (уже есть компания с таким email)	-> 409 (Conflict) "Компания с таким email уже зарегистрирована"
При обновлении — переданы несуществующие поля {"foo": "bar"} -> 400 "Поле 'foo' не поддерживается"
2. Certificate (выпуск / создание)
company_id — не существует 999	-> 404 "Компания с id=999 не найдена (или неактивна)"
recipient_name — пустая строка	"" -> 400 "Имя получателя не может быть пустым"
certificate_number — пустой или null "" -> 400 "Номер сертификата обязателен"
certificate_number — дубликат в рамках компанииномер ABC-123 уже есть -> 409 "Сертификат с номером 'ABC-123' уже существует для этой компании"
issue_date — в будущем 2030-01-01 -> 400 "Дата выдачи не может быть позже текущей даты"
issue_date — неверный формат "01-01-2025" -> 400 "Дата должна быть в формате YYYY-MM-DD"
Отсутствует PDF-файл (загрузка) -> 400 "Файл сертификата не приложен"
PDF — невалидный (битый, не PDF) текстовый файл с именем .pdf -> 400 "Загруженный файл не является корректным PDF"
PDF — не содержит извлекаемого текста	пустые страницы или скан-копия без OCR -> 400 "PDF не содержит текста. Используйте текстовая версия."
Размер PDF > 10 МБ -> 413 "Файл слишком большой (макс. 10 МБ)"
recipient_email — неверный формат (если добавили) "ivan@" -> 400 "Укажите корректный email получателя"
3. VerificationRequest (запрос на проверку)
Поведение после невалидного PDF: система не создаёт VerificationRequest (или создаёт, но сразу помечает status='no_match'? Лучше не создавать, а вернуть ошибку синхронно).
5. CRUD-операции Update для Company и Certificate
PUT /companies/{id} - Переданы поля, которые нельзя менять (например, created_at) -> 400
PUT /certificates/{id} - Пытаются изменить pdf_path или hash_sha256 (только для админа) -> 403 Forbidden
PUT /certificates/{id} - Пытаются изменить certificate_number на уже существующий в компании -> 409
PUT /certificates/{id} - Сертификат уже был участником проверки (есть связанные VerificationRequest) - изменение полей может нарушить историю -> 409
•	Уникальность certificate_number в пределах одной компании – обязательна.
•	При отзыве (revoke) – если сертификат уже revoked, повторный revoke возвращает 409 или просто ничего не делает.
•	Валидация медиа-типа – при загрузке проверять Content-Type: application/pdf.
▸ Что при конфликте данных или состояний?
1.	Ссылочная целостность (попытка удалить/изменить объект, на который ссылаются)
•	Удаление компании (DELETE /companies/{id})
o	Конфликт: у компании есть хотя бы один Certificate
o	Решение: Каскадное удаление
•	Удаление сертификата (DELETE /certificates/{id})
o	Конфликт: есть VerificationRequest, где matched_certificate_id указывает на этот сертификат, или есть Alert с certificate_id.
o	Решение: Каскадное удаление
•	Изменение certificate_number или recipient_name у сертификата, когда на него есть VerificationRequest (история проверок)
o	Конфликт: изменение «стерло бы» историю: подделка была найдена по старым данным.
o	HTTP: 409 Conflict.
o	Сообщение: «Нельзя изменить данные сертификата, так как он уже использовался в проверках. Создайте новый сертификат и при необходимости отзовите этот».
2.	Конфликт состояния (операция запрещена в текущем статусе)
•	Company is_active = false неактивная компания не должна выпускать сертификаты -> 409 «Компания заблокирована, выпуск сертификатов невозможен»
▸ Что при повреждённых данных в хранилище?
1.	Повреждение файлов PDF (оригинал сертификата или подозрительный документ)
•	Битовые ошибки на диске, сбой записи, обрезка файла, случайное перезаписывание.
•	Файл существует, указан в БД, но не открывается или не соответствует ожидаемому формату.
Как система может обнаружить:
1.	При попытке доступа (скачивание, проверка подозрительного PDF, повторное вычисление хеша или вектора).

Сценарий	Действие	HTTP / ответ пользователю	Логирование
При GET /certificates/{id}/download	Проверка существования файла, чтение. Ошибка → 500	«Файл сертификата повреждён, обратитесь к администратору»	ERROR: «Certificate {id}: PDF file corrupted or missing»
При POST /verify (подозрительный PDF)	Загруженный файл сохраняется, но при попытке чтения для извлечения текста возникает ошибка	422 Unprocessable Entity: «Загруженный файл повреждён и не может быть прочитан»	WARNING
Фоновый пересчёт хеша (административная задача)	Обнаружено несовпадение → помечаем Certificate.status = 'damaged' (новый статус) или просто логируем и отправляем админу email	Пользователю при попытке скачать: см. выше	CRITICAL: «Data integrity error: certificate {id} hash mismatch»
Восстановление:
•	Если есть резервная копия PDF – восстановить и пересчитать хеш и вектор.
•	Если резервной копии нет – сертификат становится недействительным. Нужно уведомить компанию: «Ваш сертификат №123 повреждён, пожалуйста, загрузите его заново или создайте новый».
•	Для подозрительных PDF – повреждённый файл удалить, а пользователю предложить загрузить снова.
2. Повреждение векторов (tfidf_vector в БД)
Как может повредиться: битовая ошибка в BLOB-поле (сериализованный numpy array или список Float). 
Обнаружение:
•	При проверке подозрительного PDF система загружает вектор из БД и пытается использовать его для вычисления косинусного расстояния.
•	Если десериализация падает с ошибкой (pickle, JSON) – вектор повреждён.
Реакция:
•	Отказаться от использования этого конкретного сертификата в поиске ближайшего соседа.
•	Вернуть 500 при попытке проверить, если повреждённых векторов много.
•	HTTP: 200 OK, log и сообщение пользователю «Часть данных повреждена, проверка может быть неполной».
Восстановление:
•	Пересчитать вектор заново из оригинального PDF (если PDF не повреждён).
•	Если PDF тоже повреждён – см. пункт 1.
3. Повреждение файлов моделей (.pkl с TF-IDF и NearestNeighbors)
Как может повредиться: файл на диске испорчен, частично записан, удалён.
Обнаружение:
•	При старте приложения (или при первом вызове verify) система пытается загрузить модели через joblib.load().
•	Ошибка загрузки (EOFError, UnpicklingError) или версия не соответствует ожидаемой.
Реакция:
•	Приложение переходит в деградированный режим:
o	Операции verify возвращают 503 Service Unavailable с сообщением «Система проверки временно недоступна, идёт восстановление».
o	Операции issue (создание сертификата) продолжают работать (они сохраняют PDF и могут накапливать данные для переобучения).
•	Администратор получает уведомление (email, лог).
Восстановление:
•	Запуск ручного переобучения через эндпоинт POST /models/retrain (перебирает все активные сертификаты, строит TF-IDF и NearestNeighbors с нуля).
•	Если переобучение падает (например, нет ни одного сертификата) – возвращать ошибку.
•	После успешного переобучения модели сохраняются, и проверки снова становятся доступны.
4. Повреждение базы данных (коррупция индексов, таблиц)
Как может повредиться: сбой питания при записи, сбой диска, ошибка файловой системы.
Обнаружение:
•	При попытке выполнить запрос БД возвращает ошибку (например, could not read block, index corruption).
•	В логах приложения – исключения от драйвера SQL.
Реакция:
•	Приложение возвращает 503 Service Unavailable на все запросы, требующие БД.
•	Запуск восстановления из реплики.
Восстановление (для учебного проекта):
1.	Остановить приложение.
2.	Восстановить БД из резервной копии (SQL dump).
5. Повреждение метаданных (связи между записями)
Пример: Certificate.company_id ссылается на несуществующую Company (нарушение ссылочной целостности – обычно БД не допустит, но если БД повреждена или были ручные правки).
Обнаружение: при джойне или загрузке объекта – ошибка ForeignKeyViolation или просто отсутствие родителя.
Реакция:
•	В выборке такие записи игнорировать или помечать как invalid.
•	При попытке отобразить – показывать сообщение «Данные повреждены, связь утеряна».
•	Логировать как ошибку.
▸ Что при timeout'е?
1. Тайм-аут соединения с БД
Сценарий: запрос к PostgreSQL выполняется дольше timeout
Причины: блокировки, большой объём данных, неоптимальный индекс.
Реакция:
•	Драйвер БД (SQLAlchemy, psycopg2) выбрасывает исключение (OperationalError, TimeoutError).
•	При обработке запроса (например, GET /certificates, POST /verify) – возвращаем HTTP 503 Service Unavailable.
•	Сообщение пользователю: «Сервис временно перегружен, попробуйте позже».
•	Логирование: ERROR с деталями тайм-аута и запроса.
Особенность поиска ближайшего соседа:
Если тайм-аут происходит при загрузке векторов или выполнении kneighbors, операция проверки должна быть прервана, а VerificationRequest переведён в статус timeout.
2. Тайм-аут файловой операции
Сценарий: Чтение или запись PDF-файла занимает слишком много времени.
Реакция:
•	Установить тайм-аут на операции с файлами
•	Если тайм-аут наступил – вернуть 500 Internal Server Error.
•	Пользователю: «Не удалось обработать файл, повторите попытку».
•	Логирование: ERROR.
Для POST /verify: файл мог загрузиться, но при извлечении текста произошёл тайм-аут. VerificationRequest создаётся со статусом no_match и ошибкой.
3. Тайм-аут внешнего API (отправка email)
Сценарий: SMTP-сервер не отвечает 10 секунд.
Реакция:
•	Отправка уведомления (Alert) происходит асинхронно (фоновая задача). Тайм-аут не должен влиять на основной ответ пользователю.
•	В фоновой задаче: перехватить тайм-аут, залогировать WARNING, пометить Alert.delivery_status = 'failed'.
•	Предусмотреть повторную попытку (retry) через некоторое время (например, через 5 минут, всего 3 попытки). Если все попытки закончились тайм-аутом – статус остаётся failed, администратор получает уведомление.
Пользователю в момент проверки возвращается 200 OK с сообщением: “Подделка найдена, уведомление будет отправлено”

▸ Что при очистке / потере хранилища?
Потеря базы данных (нет бэкапа)
•	Все сущности (Company, Certificate, VerificationRequest, Alert) теряются.
•	Реакция: система полностью неработоспособна. Приложение может стартовать, но все запросы будут падать с ошибками подключения.
•	Восстановление: из резервной копии (если есть).
Потеря файлов PDF (оригиналы сертификатов)
•	Записи в БД остаются, но pdf_path указывает на несуществующие файлы.
•	Реакция: при попытке скачать или использовать сертификат — ошибка 500 (или 404).
•	Фоновый процесс целостности помечает Certificate.status = 'damaged'.
•	Восстановление: если есть резервная копия файлов — восстановить. Если нет — сертификаты считаются утерянными, компания должна их перевыпустить.
Потеря файлов подозрительных PDF (VerificationRequest)
•	Записи в БД остаются, suspicious_file_path ведёт в никуда.
•	При попытке просмотреть результат проверки (например, скачать файл) — ошибка.
•	Восстановление невозможно. Пометить как corrupted.
Потеря файлов моделей (.pkl)
•	Приложение не может выполнять операцию verify (поиск ближайшего соседа).
•	Возвращает 503 Service Unavailable.
•	Восстановление: администратор запускает переобучение (POST /models/retrain), которое пересоздаёт модели на основе имеющихся активных сертификатов (если они есть).
▸ Что при обновлении схемы данных? (добавили поле — старые записи без него)
Стратегия миграции для учебного проекта: использовать инструменты миграций
Сценарий	Решение
Добавлено новое поле с значением по умолчанию	Миграция добавляет колонку, для старых записей проставляется DEFAULT.
Добавлено поле без значения по умолчанию, но nullable	Старые записи получают NULL. Код должен обрабатывать NULL
Добавлено поле NOT NULL без дефолта	Миграция сначала добавляет колонку как nullable, заполняет данные, затем меняет на NOT NULL.
Удаление поля	Миграция удаляет колонку; данные теряются.
Изменение типа поля	Миграция преобразует данные (например, VARCHAR → TEXT). Может требовать временной колонки.

▸ Какие сценарии должны работать без ошибок? (список 5-7 основных)
1.	Регистрация компании
o	POST /companies с валидным name и email -> создаётся запись, возвращается 201 Created и id.
2.	Выпуск сертификата (загрузка оригинального PDF)
o	Компания загружает PDF-файл, указывает recipient_name, certificate_number, issue_date.
o	Система вычисляет hash_sha256, извлекает текст, строит TF-IDF вектор, сохраняет сертификат.
o	Возвращается 201 Created и id сертификата.
3.	Проверка подозрительного PDF (успешное нахождение подделки)
o	Пользователь (любой) загружает подозрительный PDF.
o	Система извлекает текст, вычисляет вектор, находит ближайший оригинальный сертификат с косинусной близостью выше порога (например, ≥0.7).
o	Создаётся VerificationRequest со статусом completed, заполняются matched_certificate_id и similarity_score.
o	Автоматически создаётся Alert (уведомление компании).
o	Ответ пользователю: 200 OK с информацией о найденном владельце и степени сходства.
4.	Проверка подозрительного PDF (нет совпадений)
o	Загружен PDF, не похожий ни на один сертификат (ближайшая близость < порога).
o	VerificationRequest получает статус no_match, matched_certificate_id = null.
o	Alert не создаётся.
o	Ответ: 200 OK, «совпадений не найдено».
5.	Проверка целостности сертификата (точная проверка по хешу)
o	Пользователь скачивает оригинальный PDF через GET /certificates/{id}/download.
o	Система отдаёт файл, который должен совпадать по хешу с сохранённым hash_sha256.
6.	Отзыв сертификата (revoke)
o	Компания отправляет POST /certificates/{id}/revoke.
o	Статус сертификата меняется на revoked.
o	При последующих проверках этот сертификат не участвует в поиске (исключается из ближайших соседей).
7.	Поведение при повреждённом файле
o	Если при загрузке PDF он битый или не содержит текста → система возвращает 422 Unprocessable Entity с понятным сообщением.
▸ Какие граничные случаи должны корректно обрабатываться?
1.	Загрузка пустого PDF (0 байт) -> 400 Bad Request, сообщение «Файл пуст»
2.	PDF без текста (скан, картинка) -> 422 Unprocessable Entity, «Невозможно извлечь текст»
3.	PDF с очень длинным текстом (>3 МБ) -> Обрезать или вернуть 413 (Payload Too Large)
4.	Сертификат с дублирующимся номером в рамках компании -> 409 Conflict
5.	Запрос на проверку того же самого PDF дважды подряд -> 409 Conflict («Этот файл уже проверяется/проверен»), либо вернуть существующий результат
6.	Поиск ближайшего соседа, когда в базе 0 сертификатов -> no_match, без ошибок, без Alert
7.	Компания с is_active = false пытается выпустить сертификат -> 403 Forbidden
8.	Обновление модели (retrain) при отсутствии сертификатов -> 400 Bad Request, «Нет данных для обучения»
9.	Перезапуск системы во время загрузки PDF -> Загрузка должна быть атомарной (сначала временный файл, потом запись в БД). При сбое файл не остаётся «висячим»
10.	Тайм-аут БД при поиске ближайшего соседа -> 503 Service Unavailable, проверка не создаётся
▸ Что НЕ надо делать? (явные запреты: не добавлять фичи сверх, не менять стек)
•	Не добавлять сущности 
•	Не заменять TF-IDF на нейросети (запрещено заданием). Не использовать sentence-transformers, transformers, нейросетевые эмбеддинги.
•	Не хранить векторы в виде плотных массивов (только разреженные или сериализованные). Не пересчитывать векторы при каждом запросе – только при создании сертификата.
•	Не удалять физически записи Certificate и VerificationRequest – только логическое удаление/отзыв.
•	Не менять порог близкости без возможности конфигурации.
